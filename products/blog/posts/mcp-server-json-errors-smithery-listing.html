MCP server.json Errors That Block Your Smithery Listing

Preflight by Skillfoundry · April 2026

Smithery processes your server.json before it processes anything else. If the file fails validation, your listing doesn't get reviewed — it gets silently dropped or returns an error page with no actionable detail. Most builders discover this after the fact.

These are the specific errors that appear most often, what Smithery is actually checking, and what to change.

1. Missing or malformed remotes block

Smithery requires a remotes array containing at least one entry with type: "streamable-http". If this block is absent or uses a deprecated transport type, the listing fails.

Failing example:

{
  "name": "io.github.you/my-server",
  "version": "0.1.0",
  "remotes": [
    { "type": "http", "url": "https://example.com/mcp/" }
  ]
}

type: "http" is not the current transport identifier. The correct value is streamable-http.

Passing:

{
  "remotes": [
    { "type": "streamable-http", "url": "https://example.com/mcp/" }
  ]
}

2. name format doesn't match Smithery's slug expectations

Smithery derives a listing slug from the name field in server.json. It expects the reverse-DNS format used by the MCP Registry: io.github.yourhandle/your-server. Names that use npm-style scopes (@scope/package), plain names (my-server), or mismatched casing fail validation or produce a slug that conflicts with an existing listing.

The name must:

  • Use the reverse-DNS prefix matching your GitHub handle: io.github.yourhandle/
  • Use lowercase-hyphenated form for the package segment
  • Match the name field in your npm package.json if you're cross-listing

3. description too short or generic

Smithery displays the description prominently in search results. Descriptions under 20 characters are rejected outright. Descriptions that describe the implementation rather than what the tool does for a user are accepted but rank poorly.

Rejected: "MCP server", "A tool.", "Does stuff."

Acceptable: "Validates your MCP server against live registry rules before you publish."

The description should answer: what does this server do for the person using it?

4. version not valid semver

Smithery accepts semver strings. Non-semver values ("v1", "1.0", "latest", "dev") fail validation. Pre-release labels ("0.1.0-beta.1") are valid semver and pass, but may be sorted below stable releases in search.

5. homepage field absent or a 404

Smithery surfaces the homepage URL prominently in your listing. If the field is missing, the listing is still created in some cases, but it fails quality gates and may not be indexed. If the URL is present but returns a non-200 status, Smithery's crawler flags it as a trust signal failure.

What passes: a reachable URL (your GitHub repo, a project page, or a simple landing page). The URL must respond with 200 at submission time.

6. smithery.yaml startCommand missing or broken

If you include a smithery.yaml, Smithery uses it to build and start your server. A missing startCommand or a command that assumes local environment variables without a configSchema entry will fail Smithery's build environment. This doesn't affect the server.json submission gate directly, but it blocks listing activation after approval.

The configSchema must document every environment variable your server requires at startup. Smithery generates a configuration UI from this schema — undocumented required variables cause runtime failures that appear as listing errors to users.

How to check before you submit

Preflight validates all of the above before you submit. Connect your MCP client to the endpoint and run check_publish_readiness with your server.json, smithery.yaml, and package.json:

curl -X POST https://skillfoundry.synaplex.ai/products/preflight/api/check \
  -H "Content-Type: application/json" \
  -d '{
    "manifest": "{...server.json contents...}",
    "smithery_yaml": "{...smithery.yaml contents...}",
    "target_directories": ["smithery"]
  }'

Each failing rule returns a specific finding with what to change. Fix the findings, re-run, verify verdict: checks_pass before submitting.
